# TOTP one-time-use enforcement

The Infisical backend lets users protect MFA-gated actions with TOTP
(time-based one-time passwords) generated by an authenticator app. The
verification path lives in
`backend/src/services/totp/totp-service.ts`, where `verifyUserTotp`
decrypts the user's stored secret and validates the submitted six-digit
code with `otplib`'s `authenticator.verify`.

`authenticator.options = { window: 1 }` is set at the top of the file: a
code is considered valid if it matches the current 30-second window or
either of its immediate neighbours. That gives the user roughly 90 seconds
of skew tolerance — and during that window, **any number of submissions of
the same code value are accepted**. A code that genuinely is "one-time"
should not be redeemable twice.

## What you need to fix

After a TOTP value passes the cryptographic verification step in
`verifyUserTotp`, atomically claim it so a subsequent submission of the
same code by the same user is rejected as if the code were invalid. The
claim must:

1. Use the **shared Redis-backed keystore** (`TKeyStoreFactory`) that is
   already used elsewhere in the backend for short-lived state. Inject it
   into `totpServiceFactory` as a new dependency named `keyStore`. Only the
   atomic *set-if-not-exists with expiry* operation
   (`setItemWithExpiryNX`) is needed — narrow the dependency type with
   TypeScript's `Pick<…>` so the factory contract states exactly that.
2. Be wired in `backend/src/server/routes/index.ts` where
   `totpServiceFactory({ … })` is constructed.
3. Use a key namespaced by both the userId and the code, so distinct users
   may legitimately submit the same code value. Add a new entry to
   `KeyStorePrefixes` (in `backend/src/keystore/keystore.ts`) named
   `UsedTotpCode` with the shape:

       (userId: string, code: string) => `used-totp-code:${userId}:${code}`

4. Use a TTL long enough to cover the full ±30 s acceptance window with
   margin (the `window:1` setting accepts codes for ~90 seconds total),
   and short enough that legitimate code re-use after that window does not
   stay blocked. Add the constant to `KeyStoreTtls` as
   `UsedTotpCodeInSeconds = 120` (seconds).
5. When the atomic claim returns "already claimed" (i.e., the key existed),
   throw `ForbiddenRequestError` with message `"Invalid TOTP"` — the same
   error shape `verifyUserTotp` already emits when the cryptographic check
   fails. Do not reveal to the caller that the failure was a replay rather
   than a forged code.

The check must run **after** the otplib verification, not before — failed
attempts must not consume claim slots. The replay-prevention path is the
only behavioural change; the existing `NotFoundError` and `BadRequestError`
contracts (no config / unverified config) are unchanged, and an invalid
code still throws `ForbiddenRequestError` exactly as it did before.

## How to verify

A unit-style harness instantiates `totpServiceFactory` with an in-memory
implementation of the keystore (just `setItemWithExpiryNX`, with Redis NX
semantics). The harness:

* generates a real `otplib` code from a known secret and confirms that the
  first call to `verifyUserTotp` resolves successfully;
* re-submits the same code and confirms the second call rejects with
  `ForbiddenRequestError`;
* asserts the keystore was called exactly once with key
  `used-totp-code:${userId}:${code}` and TTL `120`;
* confirms a different user submitting the same code value is unaffected;
* confirms `KeyStorePrefixes.UsedTotpCode` and
  `KeyStoreTtls.UsedTotpCodeInSeconds` exist with the documented shape and
  a TTL between 90 and 300 seconds inclusive;
* confirms the pre-existing error contracts (`NotFoundError` for missing
  config, `BadRequestError` for unverified config, `ForbiddenRequestError`
  for an invalid code) are preserved and that an invalid code does **not**
  call into the keystore.

## Code Style Requirements

Match the existing TypeScript style in the surrounding files:

* the imports added to `totp-service.ts` must come from
  `@app/keystore/keystore` (no relative imports across the `services` →
  `keystore` boundary),
* the new keystore dependency must be typed with
  `Pick<TKeyStoreFactory, "setItemWithExpiryNX">` and added to
  `TTotpServiceFactoryDep` next to the other factory dependencies,
* the new `KeyStorePrefixes.UsedTotpCode` entry returns an `as const`
  template literal like its neighbours.

The repo enforces these via the existing `tsc --noEmit` (`npm run
type:check`) and ESLint config (`npm run lint:fix`); the test harness does
not invoke those commands, but a contradictory style would be visible in
review. No new third-party dependencies are required.
