# `Stream.decodeText` corrupts multi-byte UTF-8 characters split across chunks

You are working in the [Effect](https://github.com/Effect-TS/effect) monorepo,
checked out at a state shortly before a bug fix landed. Your job is to fix a
correctness bug in `Stream.decodeText` (exported from the `effect` package as
`effect/Stream`), then make sure the repo's existing checks remain green.

## Repository layout

The codebase lives at `/workspace/effect`. The package you will be editing is

`packages/effect/test/`. Use `pnpm` as the package manager.

## The bug

`Stream.decodeText(encoding?)` is supposed to take a stream of `Uint8Array`
chunks and produce a stream of decoded strings using a `TextDecoder`. The
operator currently throws away the decoder's internal buffer between chunks,
which is wrong: a single multi-byte UTF-8 character can be split across two
adjacent chunks, and when that happens the decoder emits Unicode replacement
characters (U+FFFD `я┐╜`) instead of the original character.

Concretely, here are scenarios that should round-trip but currently do not:

1. **A 4-byte UTF-8 character split across two chunks.** The earth-globe
   emoji `ЁЯМН` (U+1F30D) encodes to four UTF-8 bytes. If those four bytes are
   delivered as a chunk of the first two bytes followed by a chunk of the
   last two bytes, decoding the stream should yield exactly `"ЁЯМН"`.

2. **A 3-byte UTF-8 character split across two chunks.** The euro sign `тВм`
   (U+20AC) encodes to three UTF-8 bytes. Splitting them as `1 + 2` between
   two chunks should still decode back to `"тВм"`.

3. **A 2-byte UTF-8 character split across two chunks.** The Latin small
   letter n with tilde `├▒` (U+00F1) encodes to two UTF-8 bytes. Splitting
   them as `1 + 1` between two chunks should still decode back to `"├▒"`.

4. **An ASCII prefix followed by a multi-byte character split mid-byte.**
   Encoding `"Hello ЁЯМН!"` as UTF-8 and splitting it into a chunk of the first
   8 bytes (the ASCII prefix `"Hello "` plus the first 2 bytes of the emoji)
   followed by a chunk of the remaining bytes should still decode back to
   exactly `"Hello ЁЯМН!"`.

In all four cases the current implementation produces strings containing one
or more replacement characters where the multi-byte character should be.

## What to change

Fix `Stream.decodeText` so that a single logical multi-byte UTF-8 character
spread across consecutive input chunks is preserved as that character in the
output stream. The fix should not change the behaviour for chunks that
already contain whole characters.

You should *not* need to change the public API of `Stream.decodeText`, the
existing round-trip test for `Stream.decodeText`/`Stream.encodeText`, or the
`Stream.encodeText` operator. Find a small, focused fix that addresses the
streaming-decode behaviour.
