# Schema.omit on Struct + index signatures: wrong AST when an optional has a default

You are working in the `effect` (Effect-TS) repository. The TypeScript source
is under `packages/effect/src/`, tests under `packages/effect/test/`.

## Symptom

Calling `Schema.omit` on a `Schema.Struct` that combines

- a property declared with `Schema.optionalWith` configured with a
  defaulting / wrapping option such as `{ default: () => … }`,
  `{ as: "Option" }`, `{ nullable: true }`, etc., **and**
- a `Schema.Record(...)` index signature

silently produces the wrong AST. Concretely, omit a key from such a struct
and from the equivalent **plain** struct (same fields without
`optionalWith`, same `Record`):

```ts
import * as S from "effect/Schema"

const schema = S.Struct(
  { a: S.String, b: S.optionalWith(S.Number, { default: () => 0 }) },
  S.Record({ key: S.String, value: S.Boolean })
)

const plain = S.Struct(
  { a: S.String, b: S.Number },
  S.Record({ key: S.String, value: S.Boolean })
)

schema.pipe(S.omit("a")).ast   // _tag: "Transformation" — WRONG
plain.pipe(S.omit("a")).ast    // _tag: "TypeLiteral"     — correct
```

The two ASTs are expected to be `deepStrictEqual` (both `TypeLiteral`s with
one index signature and one property signature for `b`). Today they are
not — `schema.pipe(S.omit("a")).ast` is a `Transformation` node and the
index signature has been silently dropped from the omitted view.

## What to investigate

`Schema.omit` decides between two code paths:

1. an **index-signature path** that returns a `TypeLiteral` carrying the
   surviving property + index signatures, used when the input has any
   index signatures, and
2. a **`pick` path** (delegating to `Schema.pick`) used otherwise.

The decision is made by inspecting the AST for index signatures via an
internal helper. That helper recursively walks well-known wrapper nodes
(`Suspend`, `Refinement`, …) but does **not** descend into the AST node
that `optionalWith` produces around the surrounding struct. As a result,
on the example above, the helper returns an empty array, `omit` takes the
`pick` path, and the resulting AST is a `Transformation` instead of the
expected `TypeLiteral` with index signatures.

For reference, the sibling helpers `getPropertyKeys` and
`getPropertyKeyIndexedAccess` already handle this wrapper node correctly —
your fix should bring the index-signature helper into line with them.

## Expected behaviour after the fix

For the snippet above:

- `schema.pipe(S.omit("a")).ast` and `plain.pipe(S.omit("a")).ast` must be
  `deepStrictEqual`.
- The resulting AST's `_tag` must be `"TypeLiteral"`.
- It must carry exactly one index signature (for the `Record`).
- The remaining property signatures must be `["b"]` (a is omitted).
- The same must hold when the optional uses `{ as: "Option" }` instead of
  `{ default: ... }` — both are members of the same family of options on
  `Schema.optionalWith` and produce the same kind of wrapping AST node.
