# Fix the Mixture-of-Experts routing in `Dots1`

## Symptom

In the `dots1` model, the Mixture-of-Experts (MoE) router selects experts that
belong to groups that were *not* chosen as the top-`topk_group` groups for the
current token. This is a routing bug: once a group is excluded, none of its
experts may be selected.

The bug only manifests when the per-expert score-correction bias
(`e_score_correction_bias`, a learned `Parameter` on the gate) is large enough
in magnitude that `router_logits.sigmoid() + e_score_correction_bias` becomes
negative for the in-group experts. In that regime the routing function picks
out-of-group experts in preference to in-group ones — flipping the entire
group-aware routing behaviour on its head.

The existing dots1 unit-test scaffolding in
`tests/models/dots1/test_modeling_dots1.py` uses `n_group=1, topk_group=1`, so
*every* expert is in the chosen group and the bug cannot trigger there. The
defect is real for production-shape configurations (`n_group > 1`,
`topk_group < n_group`, non-trivial bias).

## Expected behaviour

For a router input of shape `(batch_size, n_routed_experts)`, the function
`Dots1MoE.route_tokens_to_experts` must:

1. Select `topk_group` groups based on the per-group score (sum of the top-2
   scores within each group of `n_routed_experts // n_group` experts).
2. Restrict `topk(..., k=top_k)` to *only* experts that lie inside the selected
   groups, even when in-group scores are negative.
3. Otherwise behave identically to `DeepseekV3MoE.route_tokens_to_experts` —
   the two routings are mathematically equivalent and were always intended to
   be so. The inline `# main diff with deepseekv3` annotation in
   `Dots1MoE.route_tokens_to_experts` is misleading; the only `diff` ever
   present was the bug.

## Repo conventions you must follow

This repository (see `AGENTS.md` / `CLAUDE.md` at the repo root) maintains
model implementations using a *modular* mechanism:

* `src/transformers/models/<model>/modular_<model>.py` is the canonical
  source. It may inherit from other models.
* `src/transformers/models/<model>/modeling_<model>.py` is **auto-generated**
  from the modular file by the converter. The header of every generated
  `modeling_*.py` says: *"Do NOT edit this file manually as any edits will be
  overwritten by the generation of the file from the modular. If any change
  should be done, please apply the change to the `modular_<model>.py` file
  directly. One of our CI enforces this."*
* `make fix-repo` (or
  `python utils/check_modular_conversion.py --files <modular file>
  --fix_and_overwrite`) regenerates `modeling_*.py` from `modular_*.py`.

After your change, the modular file and the generated modeling file must
remain consistent — the verifier runs the modular consistency check.

## Files in scope

* `src/transformers/models/dots1/modular_dots1.py`
* `src/transformers/models/dots1/modeling_dots1.py`
* `src/transformers/models/deepseek_v3/modeling_deepseek_v3.py` *(read-only
  reference — do not modify)*

## Out of scope

You do *not* need to add tests, change configs, or touch any model other than
`dots1`. The DeepseekV3 implementation is correct and is the reference.
