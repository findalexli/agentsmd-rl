# Task: Implement Warning for Unexhaustive When Over Java-Annotated Types

## Problem Description

The Kotlin FIR compiler currently does not warn users when a `when` expression over a subject type that comes from Java with nullability annotations (e.g., `@Nullable`) could potentially be null, but the `when` expression lacks a null or else branch.

### What's Broken

When a Kotlin `when` expression is exhaustive over a sealed class or enum, but the subject comes from a Java method annotated with nullability annotations (like JSpecify's `@Nullable`), the compiler should warn that the `when` might not handle the null case. Currently, no such warning is emitted.

### Example of Missing Warning

```kotlin
// Java file with JSpecify annotations
import org.jspecify.annotations.*;

public class JavaClass {
    @Nullable
    public static String getNullable() { return null; }
}

// Kotlin code that should trigger a warning but doesn't
sealed class Result
object Success : Result()
object Failure : Result()

fun test(r: Result) {
    // This when is exhaustive over Result, but JavaClass.getNullable()
    // could return null at runtime due to @Nullable annotation
    when (JavaClass.getNullable() as Result) {
        is Success -> {}
        is Failure -> {}
        // Missing null branch - should warn!
    }
}
```

The compiler should emit a warning like:
```
'when' expression over a subject of type 'Result' is not exhaustive. Add a 'null' or 'else' branch.
```

## Task

Implement a new FIR checker that:

1. Detects when a `when` expression is over a subject whose type has an "enhanced" type for warnings (from Java nullability annotations)
2. Checks if the enhanced type can be null (but the original type cannot)
3. Warns if there's no null check branch in the `when` expression

## Implementation Hints

1. The checker should extend `FirWhenExpressionChecker` with `MppCheckerKind.Platform`
2. Use `expression.isExhaustive` to check if the when is already exhaustive
3. Use `enhancedTypeForWarning` to get the type enhanced with Java nullability info
4. Check if the enhanced type can be null but the original type cannot
5. Look for null checks in branches via `FirEqualityOperatorCall` or `FirTypeOperatorCall`
6. Report `UNEXHAUSTIVE_WHEN_BASED_ON_JAVA_ANNOTATIONS` diagnostic if no null check found
