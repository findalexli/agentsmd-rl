# Add a per-RPC defect schema option to `@effect/rpc`

You are working in the [Effect-TS/effect](https://github.com/Effect-TS/effect)
monorepo. The package of interest is `@effect/rpc`
(`packages/rpc/`).

## Background

In `@effect/rpc`, when a server-side handler dies (i.e. produces a defect via
`Effect.die`), that defect must cross the RPC boundary back to the client.
Today the library hard-codes `Schema.Defect` for this serialization on every
RPC. `Schema.Defect` is built around the `Error` shape — it preserves
`name`/`message`/`stack`, but for arbitrary non-`Error` defects it round-trips
through a string representation, losing structured fields.

For internal service boundaries, callers want richer fidelity: the ability to
say "for *this* RPC, serialize the defect with a schema I control" — for
example `Schema.Unknown`, which preserves the value as-is.

## Your task

Add an **optional** `defect` parameter to `Rpc.make` that lets a caller
override the defect schema for a single RPC. When omitted, behavior is
unchanged (defaults to `Schema.Defect`). The chosen schema must flow through
to `Rpc.exitSchema` and the server-side defect-encoding path so that the
custom serialization is actually used end-to-end.

## Required public surface

After your change, this must compile and run correctly:

```ts
import { Rpc } from "@effect/rpc"
import { Schema } from "effect"

const myRpc = Rpc.make("MyRpc", {
  success: Schema.String,
  defect: Schema.Unknown
})
```

### Behavioral requirements

1. **`rpc.defectSchema` property.** The constructed `Rpc` instance must
   expose its defect schema as a property named `defectSchema`. When the
   `defect` option is provided, `rpc.defectSchema` must be the exact same
   schema reference passed in (`assert.strictEqual` equality). When the
   `defect` option is omitted, `rpc.defectSchema` must still be populated
   with the existing default (`Schema.Defect`) — never `undefined`.

2. **`Rpc.exitSchema(rpc)` must use `rpc.defectSchema`.** Currently
   `exitSchema` hard-codes `Schema.Defect` for the defect slot of the
   returned `Schema.Exit({...})`. After your change, it must use the
   per-RPC defect schema. As a concrete consequence, the following must
   round-trip identically:

   ```ts
   const rpc = Rpc.make("X", { success: Schema.String, defect: Schema.Unknown })
   const schema = Rpc.exitSchema(rpc)
   const original = { code: 99, detail: "ohno", deep: { x: 1 } }
   const encoded = Schema.encodeSync(schema)(Exit.die(original))
   const decoded = Schema.decodeSync(schema)(encoded)
   // Cause.squash of the decoded failure must deepStrictEqual `original`.
   ```

   With the current `Schema.Defect` hard-coding the decoded value comes back
   as the JSON string `'{"code":99,"detail":"ohno","deep":{"x":1}}'`, not the
   structured object — that is the bug.

3. **No regressions.** Every other `Rpc` consumer must keep working. In
   particular, the field must be threaded through every existing `Rpc`
   construction site and through every `Proto` method that already threads
   `successSchema` / `errorSchema` (e.g. the methods that build a new `Rpc`
   when adding middleware, error schemas, payload, annotations, etc.). The
   server-side encoding path (`packages/rpc/src/RpcServer.ts`) must also
   honor the per-RPC schema when emitting `Exit` / `Chunk` failures and
   request-decode defects.
