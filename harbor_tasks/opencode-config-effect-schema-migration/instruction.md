# Migrate three leaf config schemas to Effect Schema

The repo (`opencode`, a TypeScript/Bun monorepo) is mid-migration from
hand-written Zod schemas to Effect Schema, with `.zod` compatibility shims
generated by `zod()` in `@/util/effect-zod`. The migration order and style
rules are documented in `packages/opencode/specs/effect/schema.md`.

You are migrating the next slice: three leaf schemas under
`packages/opencode/src/config/`, plus the two consumers that already call
into them.

## What is broken right now

The schemas in `packages/opencode/src/config/skills.ts`,
`packages/opencode/src/config/formatter.ts`, and
`packages/opencode/src/config/console-state.ts` are still defined with `zod`
directly. They do not expose the `.zod` accessor that the rest of the
migrated codebase has adopted, and they cannot participate in any future
Effect-Schema-first DTO surface.

Concretely, the failing contract is:

- `ConsoleState.zod.safeParse({...})` should accept and reject payloads
  matching the existing shape `{ consoleManagedProviders: string[],
  activeOrgName?: string, switchableOrgCount: number }`. Today
  `ConsoleState.zod` is `undefined`.
- `ConsoleState.make({...})` should construct an instance (Schema.Class
  semantics). Today the export is a plain Zod object that has no `.make`.
- `ConfigSkills.Info.zod.safeParse({...})` should validate the
  `{ paths?: string[], urls?: string[] }` shape. Today `.zod` is `undefined`.
- `ConfigFormatter.Info.zod.safeParse(...)` should validate either
  `boolean` or `Record<string, Entry>`, and `ConfigFormatter.Entry.zod`
  should validate the `{ disabled?, command?, environment?, extensions? }`
  entry shape. Today both `.zod` accessors are `undefined`.

The two existing consumers that touch these schemas are also wrong:

- `packages/opencode/src/config/config.ts` calls `ConfigSkills.Info.optional()`
  and `ConfigFormatter.Info.optional()` directly — that only works while those
  exports are Zod schemas. After migration these calls must read the `.zod`
  shim instead. (For comparison, the same file already uses
  `ConfigLSP.Info.zod.optional()` for an already-migrated sibling — match that
  pattern.)
- `packages/opencode/src/server/routes/instance/experimental.ts` passes
  `ConsoleState` directly into Hono's `resolver(...)`, which expects a Zod
  schema. After migration it must pass the `.zod` shim.

## What "fixed" looks like

Migrate each of the three schema files following the rules in
`packages/opencode/specs/effect/schema.md`, the schema/style rules in
`packages/opencode/AGENTS.md`, and the existing already-migrated
`config/lsp.ts` / `config/agent.ts` etc. as worked examples. In particular:

- **`console-state.ts` is exposed by an endpoint** (the `/experimental/console`
  route's 200 response) — it should be a *named* schema with a `$ref` in
  OpenAPI. Use the `Schema.Class<Self>("Name")({ ... })` form with
  `static readonly zod = zod(this)`. Preserve the existing
  `emptyConsoleState` constant.
- **`skills.ts` and `formatter.ts` are only nested inside the root `Config.Info`** —
  they should stay anonymous. Use the `Schema.Struct({...}).pipe(withStatics((s)
  => ({ zod: zod(s) })))` form with a separately-declared `type Info =
  Schema.Schema.Type<typeof Info>`. Adding a named ref to a nested-only schema
  would bloat the generated SDK with no gain.
- The `withStatics` helper lives at `@/util/schema`; the `zod()` converter
  lives at `@/util/effect-zod`. Use these — do not write a parallel Zod
  definition by hand.
- `Schema.Array(Schema.String)` produces a `ReadonlyArray`. Wrap with
  `Schema.mutable(...)` where the existing type signature is `string[]`
  (the formatter's `command`/`extensions` and the console-state's
  `consoleManagedProviders` are the relevant cases).
- `Schema.Record` is positional: `Schema.Record(Schema.String, ValueSchema)`.
- `Schema.Union` takes an array: `Schema.Union([A, B])`.
- The `.describe(...)` calls on the old Zod fields should become
  `.annotate({ description: "..." })` on the corresponding Effect Schema
  fields.
- Preserve each file's existing self-reexport at the top or bottom (e.g.
  `export * as ConfigSkills from "./skills"`). Do not introduce
  `export namespace Foo { ... }` — flat top-level exports plus self-reexport
  is the project's module shape.

Then update the two consumers to read `.zod` instead of treating the migrated
exports as Zod schemas directly.

## Code Style Requirements

The repo's `AGENTS.md` style guide applies to every edited file:

- Prefer `const` over `let`.
- Avoid `else` after an early `return`.
- Avoid the `any` type.
- Avoid unnecessary destructuring; use dot notation.
- Rely on type inference; avoid explicit annotations unless required for an
  exported boundary.

There is no separate linter step in this task, but the rubric will score on
adherence.

## Out of scope

- The auto-generated SDK types in `packages/sdk/js/src/v2/gen/types.gen.ts`
  are regenerated by a separate build script; you do not need to update them
  for this task.
- Other config schemas (`mcp.ts`, `permission.ts`, `agent.ts`, root
  `config.ts#Info`) are tracked as follow-ups. Do not migrate them here.
