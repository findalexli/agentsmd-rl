# Task: Fix three bugs in the transformers library at commit eb3d67aaafe368863afc77e4b60fa60c2077c8b5

## Bug 1: Config serialization excludes per-instance fields

`transformers_version` and `architectures` are fields on the base configuration class that should be included in serialization. However, these fields are currently declared in a way that makes them class-level rather than instance-level, which excludes them from `dataclasses.fields()` (making them invisible to serialization logic). They are also absent from the config attribute validation allowlist. Each instance should maintain its own values for these fields.

**Symptom**: `transformers_version` and `architectures` do not appear in `dataclasses.fields()` of the base config class. Setting `config1.transformers_version = "1.0"` does not isolate it from `config2.transformers_version = "2.0"`.

**Expected behavior**:
- `dataclasses.fields()` of the base config class must include both `transformers_version` and `architectures`
- Per-instance isolation: `c1.transformers_version = "1.0.0"` and `c2.transformers_version = "2.0.0"` must not affect each other; same for `architectures`
- The config attribute validation allowlist must include both `transformers_version` and `architectures`
- The `config_common_kwargs` completeness check in the test suite expects a specific set of missing keys. After the fix, the keys present in `PreTrainedConfig().__dict__` but absent from `config_common_kwargs` must be exactly, in this order: `["transformers_version", "is_encoder_decoder", "tokenizer_class", "_name_or_path", "_commit_hash", "_output_attentions", "_attn_implementation_internal", "_experts_implementation_internal"]`

## Bug 2: RoPE validation crashes on empty dict

The `RotaryEmbeddingConfigMixin` class has a rope validation method that crashes when a config has `rope_parameters = {}` (empty dict) instead of `None`. The method handles the `None` case but does not handle the empty dict case.

**Symptom**: Creating a config with `rope_parameters = {}` causes the validation method to raise an error or mutate the dict. The same config with `rope_parameters = None` works fine.

**Expected behavior**:
- Calling the validation method with `config.rope_parameters = {}` must not crash
- `rope_parameters` must remain `{}` after the call (not mutated)
- Multiple sequential configs with empty dict must all work correctly
- The `None` case must still work (anti-regression)
- When `rope_parameters` contains a `"rope_type"` key with a value like `"test_dispatch"` (e.g., `{"rope_type": "test_dispatch", "factor": 1.0}`), the method must dispatch to the corresponding per-type validation method following the naming convention `_validate_<type>_rope_parameters`. The dispatch must actually occur (the method must not be a no-op stub).

## Bug 3: Docstrings include inherited parameters for config classes

Config classes have their docstrings generated by an auto-docstring module. The functions `_process_regular_parameters`, `_process_parameters_section`, and `auto_method_docstring` currently include ALL parameters from the class hierarchy, including inherited ones. They need to accept a filtering mechanism so that only parameters defined directly on the class are included.

**Symptom**: Config subclass docstrings include parameters from parent classes that the subclass didn't actually define.

**Expected behavior**:
- `_process_regular_parameters`, `_process_parameters_section`, and `auto_method_docstring` must each accept an `allowed_params` keyword argument
- When `allowed_params={"alpha", "gamma"}` is passed to `_process_regular_parameters`, parameters not in the set (e.g., `"beta"`) must be filtered out from the result
- Without `allowed_params`, all parameters must be included (no filtering)

## Requirements

1. `transformers_version` and `architectures` must appear in `dataclasses.fields()` of the base config class and behave as instance fields with per-instance isolation.
2. The RoPE validation method must not crash when `rope_parameters` is an empty dict `{}` and must not mutate the dict.
3. Docstring generation for config classes must support filtering to only include specific parameters via an `allowed_params` keyword argument on the relevant functions.
4. The config attribute validation allowlist must include `transformers_version` and `architectures`.
5. All modified files must pass syntax, ruff format, and ruff lint checks.
