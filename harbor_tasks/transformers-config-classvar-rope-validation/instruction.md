# Task: transformers-config-classvar-rope-validation

Fix three bugs in the transformers library at commit eb3d67aaafe368863afc77e4b60fa60c2077c8b5:

## Bug 1: Config serialization excludes per-instance fields

`transformers_version` and `architectures` are fields on `PreTrainedConfig` instances that should be included in serialization. However, these fields are currently excluded from `dataclasses.fields()` (making them invisible to serialization logic) and are not included in the config attribute validation allowlist. Each instance should maintain its own values for these fields.

**Symptom**: `transformers_version` and `architectures` do not appear in `dataclasses.fields(PreTrainedConfig)`. Setting `config1.transformers_version = "1.0"` does not isolate it from `config2.transformers_version = "2.0"`.

**Tests verify**:
- `dataclasses.fields(PreTrainedConfig)` includes `transformers_version` and `architectures`
- Per-instance isolation: `c1.transformers_version = "1.0"` and `c2.transformers_version = "2.0"` do not affect each other
- `ATTRIBUTES_TO_ALLOW` in `utils/check_config_attributes.py` includes both fields
- The `config_common_kwargs` completeness check in `tests/utils/test_configuration_utils.py` expects `transformers_version` in the missing keys list

## Bug 2: RoPE validation crashes on empty dict

The `validate_rope` method on `RotaryEmbeddingConfigMixin` crashes when a config has `rope_parameters = {}` (empty dict) instead of `None`. The method only checks `if rope_parameters_dict is None` but does not handle the empty dict case.

**Symptom**: Creating a config with `rope_parameters = {}` causes `validate_rope` to raise an error or mutate the dict. The same config with `rope_parameters = None` works fine.

**Tests verify**:
- `validate_rope(config)` where `config.rope_parameters = {}` does not crash
- `rope_parameters` remains `{}` after the call (not mutated)
- Multiple sequential configs with empty dict all work correctly
- The `None` case still works (anti-regression)

## Bug 3: Docstrings include inherited parameters for config classes

Config classes (dataclasses) have their docstrings generated by `auto_class_docstring`, which currently includes ALL parameters from the class hierarchy, including inherited ones from `PreTrainedConfig`. This produces overly long docstrings.

**Symptom**: Config subclass docstrings include parameters from parent classes that the subclass didn't actually define.

**Tests verify**:
- `_process_regular_parameters`, `_process_parameters_section`, and `auto_method_docstring` accept an `allowed_params` keyword argument
- When `allowed_params={"alpha", "gamma"}` is passed, parameters not in the set are filtered out
- Without `allowed_params`, all parameters are included

## Requirements

1. `transformers_version` and `architectures` must appear in `dataclasses.fields(PreTrainedConfig)` and behave as instance fields with per-instance isolation.
2. `validate_rope` must not crash when `rope_parameters` is an empty dict `{}` and must not mutate the dict.
3. Docstring generation for config classes must only include parameters defined directly on the class, not inherited parameters.
4. The config attribute validation allowlist must include `transformers_version` and `architectures`.
5. All modified files must pass syntax, ruff format, and ruff lint checks.
