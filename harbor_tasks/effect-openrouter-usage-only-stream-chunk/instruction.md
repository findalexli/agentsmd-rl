# Fix OpenRouter streaming finalization for usage-only terminal chunks

You are working in the `Effect-TS/effect` monorepo. The repo is checked out at
`/workspace/effect`. The package of interest is `@effect/ai-openrouter` under
`packages/ai/openrouter/`.

## The bug

`@effect/ai-openrouter` consumes a Server-Sent-Events stream from the
OpenRouter chat-completions API and turns each parsed chunk into output
parts (`text-delta`, `finish`, etc.) for the higher-level `LanguageModel`
service.

OpenRouter recently changed how it terminates a streamed response. The
final SSE chunk can now contain *only* a populated `usage` block and an
empty `choices` array, e.g.:

```json
{
  "id": "...",
  "model": "openai/gpt-4",
  "choices": [],
  "usage": {
    "prompt_tokens": 13,
    "completion_tokens": 16,
    "total_tokens": 29
  }
}
```

The streaming adapter in
`packages/ai/openrouter/src/OpenRouterLanguageModel.ts` currently rejects
any chunk whose `choices` array is empty as malformed. As a result, every
streamed response now fails at the very end with an
`AiError.MalformedOutput` whose description is
`"Received response with no valid choices"` -- even though the model
already streamed all of its content successfully.

## What "fixed" looks like

The streaming adapter must accept a terminal chunk that has no choices
**only when** that chunk carries a `usage` block. In that case the stream
should complete normally and emit a `finish` part populated from the
`usage` data:

- `usage.inputTokens` ŌåÉ `event.usage.prompt_tokens`
- `usage.outputTokens` ŌåÉ `event.usage.completion_tokens`
- `usage.totalTokens` ŌåÉ `event.usage.total_tokens`

A chunk that has *neither* a choice *nor* a `usage` block is still
malformed and must continue to raise `AiError.MalformedOutput` with the
same `"Received response with no valid choices"` description -- the fix
is targeted at the usage-only case, not a wholesale relaxation.

Existing behaviour for non-terminal chunks (text deltas, tool calls,
reasoning parts, etc.) must be preserved: an upstream chunk that carries
text content followed later by a usage-only terminator should produce the
same `text-start` / `text-delta` / `text-end` parts as before, plus the
new `finish` part.

## Where to look

The relevant function is `makeStreamResponse` in
`packages/ai/openrouter/src/OpenRouterLanguageModel.ts`. It takes a
`Stream.Stream<ChatStreamingResponseChunk, ...>` and produces a
`Stream.Stream<Response.StreamPartEncoded, ...>`. Whichever shape your
fix takes, the function as a whole must remain correct on every existing
input shape -- text deltas, tool-call deltas, reasoning deltas, source
annotations, image deltas, and `finish_reason` chunks -- in addition to
the new usage-only terminal case.

The `ChatStreamingResponseChunk` schema is defined in
`packages/ai/openrouter/src/OpenRouterClient.ts` -- both `choices` and
`usage` are present there as optional fields, with `usage` typed as
optional/nullable.
