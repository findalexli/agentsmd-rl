# Generalize the vision-mask helper to support arbitrary group ids

Several multimodal models in `transformers` define a small helper called
`token_type_ids_mask_function` that builds an attention-mask hook used by
`create_causal_mask_mapping`. The current implementation accepts two
tensors â€” `token_type_ids` and `image_group_ids` â€” and only unmasks
attention between two positions when both are images
(`token_type_ids == 1`) belonging to the same image group.

We want to generalize this helper so it can describe any kind of vision
token (images today, videos in the future), not just images. Refactor
the helper so it has the contract described below, in every model where
it lives.

## New contract

`token_type_ids_mask_function` must:

1. Accept **exactly one positional argument** named `group_ids`. It is a
   `(batch_size, seq_len)` integer tensor where each entry is either
   `-1` (text â€” must not produce extra unmasking) or a **non-negative
   integer** identifying the vision group that token belongs to. Tokens
   with the same group come from the same input image (or, in the
   future, the same video).

2. Return a callable `inner_mask(batch_idx, head_idx, q_idx, kv_idx)`
   that returns `True` exactly when the q-token and the kv-token are in
   the **same non-negative group**, and `False` in every other case â€”
   including textâ†”text, textâ†”vision, visionâ†”text, and any two vision
   tokens that belong to different groups.

3. Remain robust to indices that exceed `seq_len`. With static caches,
   the kernel can call `inner_mask` with `q_idx` or `kv_idx` larger than
   `group_ids.shape[-1]`. In that case the function must return `False`
   without raising any indexing error.

4. Treat each batch row independently â€” masking decisions for batch
   `b` must only consult row `b` of `group_ids`.

## Where to apply this

The helper is defined (and called from `create_causal_mask_mapping`) in
three model implementations: **gemma3**, **git**, and **paligemma**.
You will need to update each definition and each call site so that
the call passes the `group_ids` tensor (built from `token_type_ids` and
the image-start mask) directly, with no second argument.

The repository keeps these three copies in sync via two mechanisms
documented in `CLAUDE.md` / `AGENTS.md`:

- A `# Copied from transformers.models.gemma3.modeling_gemma3.token_type_ids_mask_function`
  marker on one of the three. The repo's consistency check
  (`python utils/checkers.py copies`) verifies the two functions remain
  byte-for-byte equivalent (modulo renames). After your edits this
  check must pass.
- A `modular_*.py` source file from which one of the `modeling_*.py`
  files is regenerated by the modular converter. Edits made only in the
  generated file are reverted by `make fix-repo`. Make sure to update
  the modular source so its call site also passes `group_ids` as a
  single argument.
